# Day 21 - Networks Are Just Conversations

## Key Learnings

- Twenty-five days, four sub-arcs: the model, the inspection, the attacks, the defences
- Sender, receiver, language, route. Every network conversation in the world has those four parts
- A network packet works almost identically. Address on the outside, payload on the inside, routed by intermediaries.
- Four attack categories: spoof the sender, flood the receiver, exploit the language, intercept the route.

  # Day 22 - IP Address

  ## Key Learnings

  - An IP address is a unique identifier assigned to a device on a network
  -  IPv4 (Internet Protocol version 4). Each of those four numbers is 8 bits, so the whole address is 32 bits
  -  Every IPv4 address has two halves: the network part (which network the device is on) and the host part (which specific device on that network)
  - Public IP addresses are unique on the entire internet, Private IP addresses are reused inside private networks
    ### NAT (Network Address Translation)
     When your laptop sends a packet to Google, your router does the following:
    
- Notes that your laptop (`192.168.1.5`) wants to talk to Google
- Replaces your laptop's private address with the router's own public address before sending the packet out

# Day 23 - Ports

## Key Learings

- A port number tells your device which application should receive it, every network conversation has both a destination IP and a destination port
- A port is a 16-bit number, which gives it a range from 0 to 65535. Every conversation has a source port (random, chosen by the OS) and a destination port (specific to the service being contacted)

• 22 — SSH (secure shell, the standard remote-login protocol for servers)
• 25 — SMTP (sending email between mail servers)
• 53 — DNS (domain-name lookups, tomorrow's lesson)
• 80 — HTTP (the unencrypted web)
• 110 — POP3 (older email retrieval)
• 143 — IMAP (modern email retrieval)
• 443 — HTTPS (the encrypted web — most of what you use)
• 587 — SMTP submission (sending email from a mail client)
• 993 — IMAPS (encrypted IMAP)
• 3306 — MySQL (database)
• 3389 — RDP (Remote Desktop Protocol — Windows remote access)
• 5432 — PostgreSQL (database)
• 5900 — VNC (Virtual Network Computing — remote desktop)

- Ports 0-1023 are the well-known ones (HTTP, HTTPS, SSH, DNS, etc.). Ports 1024-49151 are registered for specific applications. Ports 49152-65535 are dynamic, used by your OS as random source ports for outgoing connections.
- Open ports are attack surface. The fewer open, the safer. The most dangerous category is "open to the public internet" because Shodan and similar scanners find them within hours
- Every open port should answer four questions: why open, what service, is it patched, who needs access
- Almost all consumer internet traffic runs on port 443 (HTTPS)
- IP + port = the complete address for a network conversation

# Day 24 - DNS

## Key Learnings

-  Every device caches answers locally for speed. When the cache misses, the recursive resolver does the lookup work on your behalf
- A DNS lookup walks a hierarchy: root → TLD (.com, .org) → authoritative server → resolver → your device. Every hop caches its answer. Every hop is also a trust point an attacker could target.
  
### Three properties of DNS make it permanently attractive to attackers:
- Most queries are still unencrypted: evealing every site you visit) and can modify the responses (sending you to a different IP). DNS-over-HTTPS (DoH) and DNS-over-TLS (DoT) fix this, but adoption is still partial
- Caching makes attacks sticky: A poisoned cache entry stays poisoned until the TTL expires, sometimes hours
- Authentication is rare in practice: DNSSEC (DNS Security Extensions) was designed to cryptographically sign DNS responses so attackers can't forge them

### DNS attacks three of the most important categories:
- DNS hijacking: The attacker takes over the DNS records for a domain and points the name to an IP they control, done by compromising the registrar account, the authoritative server, or the resolver's upstream
- DNS cache poisoning: The attacker injects a malicious answer into a recursive resolver's cache, so every user of that resolver gets the wrong IP until the cache expires
- DNS rebinding: A subtler attack where a malicious site exploits the fact that DNS answers can change quickly. After the initial page loads, the DNS record is updated to point to an internal IP (like a router admin panel), letting the malicious page reach inside the user's network


- DNS sits on top of BGP. Compromise BGP, and you can compromise DNS without ever touching the DNS infrastructure itself.
- MyEtherWallet shows DNS attacks at internet scale: BGP hijack to redirect DNS traffic, fake DNS server to redirect users, fake TLS-less site to capture credentials. The defences (DNSSEC, HSTS, encrypted DNS) all existed
- The defences are tiny — registrar lock, MFA on the registrar, reduced public footprint — but the cost of skipping them is "your entire online presence rerouted to a fake site."
- Every time, the attack goes through the registrar account, not the domain itself.

## Commands used:
- nslookup wikipedia.org 1.1.1.1

# Day 25 - TCP v/s UDP

## Key learnings

- TCP is reliable, ordered, and slow. UDP is unreliable, unordered, and fast
- Every TCP conversation begins with a three-way handshake:

• The client sends a SYN ("synchronise") packet. This says: I want to start a conversation.
• The server replies with a SYN-ACK packet. This says: I'm ready, and I acknowledge your request.
• The client replies with an ACK packet. This says: Acknowledged. We're connected.

- Three packets, each tagged with a sequence number, Every packet is sequenced and acknowledged
- TCP also handles flow control (slowing the sender down when the receiver can't keep up) and congestion control (slowing both sides down when the network is overloaded
- UDP sends and forgets. No handshake, no sequencing, no retransmission. Used by DNS, NTP, voice, video, gaming, and DHCP
- SYN Floods(TCP): The TCP handshake requires the server to remember every half-open connection until the client sends the final ACK. An attacker can exploit this by sending thousands of SYN packets and never sending the final ACK
- UDP amplification (UDP): Because UDP has no handshake, the source IP on a UDP packet is trivially spoofable. An attacker sends a small UDP query to a public server, but with the source IP forged to point at the victim. DNS, NTP, and Memcached are the three most-abused amplifiers because they each have small-request-large-response patterns
- UDP floods (UDP): Simpler than amplification: just blast UDP packets at every port of a target. The target's OS spends CPU cycles checking whether any service is listening
- Connection hijacking (TCP): If an attacker can predict a TCP sequence number and is on the network path, they can inject packets that look legitimate to both ends and steal the connection

# Day 26 - Firewalls

## Key Learnings

- A firewall is a control that allows or blocks network traffic based on rules
- Four firewall categories from simplest to most capable: packet-filtering (each packet alone), stateful (tracks conversations), proxy (acts as middleman), next-generation (inspects application content)
- Firewall rules are ordered. The first match wins
- The catch-all deny at the bottom is non-negotiable in mature configurations. It's called default deny or implicit deny, and it implements a fundamental security principle: only explicitly allowed traffic is permitted; everything else is blocked
- Whitelist (or allowlist) is the model used above. Explicitly permit what's allowed, deny everything else
- Blacklist (or blocklist) is the inverse. Explicitly block known-bad, allow everything else. Lower security, less administrative effort
- Firewalls sit at multiple points: perimeter (network edge), internal segmentation (between zones), host-based (on each device), cloud (provider-managed), and WAF (application layer). Modern security uses several of these together — defence in depth — rather than relying on one perimeter firewall
- Firewalls don't stop attacks within allowed traffic, attacks within encrypted traffic, social engineering, insider threats, or outbound exfiltration without specific configuration. They're one control in a layered defence — necessary, not sufficient
